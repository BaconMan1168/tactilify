A — Layered patch

Extend the existing planner and renderer in place. Add symbolHint/explorationInstructions to the schema, wire domain symbol logic into each layout function, restructure page zones, bolt on split logic.

Pros

  • Minimal new files
  • Least disruption to working code

Cons

  • Planner grows very large
  • Domain logic mixed with layout logic
  • Harder to test domains independently

B — New adaptor stage (Recommended)

Insert a tactileAdaptor.ts between analysis and planner. It classifies the domain, maps elements to tactile symbols, and decides whether/how to split pages. The planner stays focused on geometry only.

Pros

  • Clean separation: domain logic vs geometry
  • Each stage testable in isolation
  • Multi-page falls out naturally (adaptor returns page specs)
  • Planner stays simple

Cons

  • New intermediate type TactileAdaptation
  • One more file in the pipeline

C — Domain registry

Replace the layout-function dispatch with a registry of domain handlers. Each domain (circuit, FBD, chemistry, chart, flowchart, generic) owns its full layout + symbol strategy. Maximally extensible.

Pros

  • Each domain fully isolated
  • Easy to add new domains later

Cons

  • Most new code
  • Overkill for 5 domain types
  • Risk of divergence between handlers
